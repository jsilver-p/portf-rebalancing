# 로컬 추출 모델 선정 — 결정 v2.2 (2026-07-16)

**결정: 구성 변경 없음 — v2.1 구성(32B+prompt4c+NP=2+상시로드) 유지. 단, 실측의 기준이 바뀐다.**

## v2.2 — 속도 2차 최적화 시도: 전 지렛대 기각, 대신 측정 기반을 교정했다

### v2.1의 35.5s/장은 캐시 오염이었다 (이번 판의 가장 중요한 교정)

`keep_alive:-1`로 살아있는 슬롯이 **직전 런과 동일한 (이미지, 프롬프트)의 KV를 prefix 재사용**한다
— 같은 픽스처를 반복 측정하는 벤치마크에서 prefill이 0.2s로 붕괴(20,000 tok/s처럼 보임).
v2.1의 284s(35.5s/장)가 이것이다. **캐시 퇴거(서로 다른 장문 2발을 NP개 동시 발사) 후 신선 실측:
497s/8 = 62s/장.** 프로덕션의 새 스크린샷은 항상 신선 = 62s/장이 진짜 기준선.
(부산물: 같은 이미지 재추출은 prefill이 거의 무료 — 재시도·재검증엔 유리한 성질.)

### 물리 재분해 (신선 기준, 장당)

| 구간 | 실측 | 성질 |
|---|---|---|
| ViT 인코딩 | ~9.6s | prompt_eval에 섞여 있음(v2.1의 "145tok/s"는 이것 포함한 혼합값) |
| LLM prefill | 4.3~5.3k tok @ **216 tok/s** = 20~24s | compute-bound, 요청 간 직렬 |
| decode | ~300 tok @ 단독 6.9 t/s, NP=2 합산 11.2 t/s | 대역폭 포화 |
| wall(NP=2) | ≈ (prefill합+decode합)/2 ≈ 62s/장 | prefill·decode **공동 병목** |

이미지 토큰 회계: HF 레퍼런스(patch 14·merge 2 → **px/784**)와 ollama가 **일치**(3276 vs 3298).
"ollama가 토큰을 4배 과다 계상한다"는 가설은 스펙 오독이었다(패치 28px 아님). 예외: 4928px
톨스크린은 ollama가 ~4k 토큰으로 캡(레퍼런스 6864) — 그런데도 parity 통과.
→ **vLLM 전환에 '공짜 토큰 절감'은 없다.** 이득은 커널 효율(추정 2~3배)뿐.

### 시도한 지렛대 — 전부 기각 (정확도 게이트에 걸림)

| 지렛대 | wall | 정확도 | 판정 |
|---|---|---|---|
| E2 지시문→system 이동(prefix KV 재사용, -1267tok/장) | 470s(-5%) | **FAIL** — 160229 '미국 달러' 환각 1 + 원화 잔고행 유실(30/31) | ❌ |
| E3 보수 크롭(상태바 90px+소프트내비 110px+가장자리 잉크트림, -9.8% px) | 481s(-3%) | **FAIL** — mPOP 병합셀 평가↔매수 스왑 7건(value 24/31) | ❌ |
| E4 7B 기본 + 게이트 경고 화면만 32B 재추출(파동식) | (207s+에스컬) | **FAIL** — USD 중복행 유실 시 합계 게이트 침묵(FN>0 실증, 아래) | ❌ |
| E5 qwen2.5vl:3b 단독 | 130s(16.3s/장) | **FAIL** — 재현율 0/31, 환각 2(출력 50~175tok로 붕괴) | ❌ |

**E4의 구조적 맹점(중요)**: 게이트는 합계 대조다. 외화예수금 화면에서 7B가 USD 표기행을 떨구고
원화 잔고행만 내면 요약-상세 합계는 여전히 맞는다 → 경고 없이 행 유실. 캐시가 균일하던 어제의
3런 "결정론적 회복(3/8장 교체 PASS)"은 착시였고, 신선 런에선 실패 화면 자체가 달라졌다(비결정).
**행 존재를 보증하지 못하는 게이트 위에 작은 모델을 얹을 수 없다.**

### 이번 판의 교훈

1. **31/31은 안정 평형이 아니라 칼날이다.** 크롭(기하), 프롬프트 위치(system/user), 캐시·배칭
   상태(수치 미세차) — 어느 미세 변화에도 특정 화면의 셀 귀속이 무너졌다. temp=0이어도
   continuous batching의 슬롯 상태에 따라 raw가 달라진다(신선 런 name 1건·price_src 8건 흔들림,
   PASS는 유지). 속도 실험은 반드시 **신선 캐시 + 8장 전체 parity**로만 판정할 것.
2. **속도 벤치마크는 캐시 퇴거가 선행 조건.** 퇴거는 서로 다른 장문을 NP개 **동시** 발사
   (순차 발사는 같은 슬롯만 두 번 덮는다).
3. 측정 하네스: `run_extract.py`가 이제 `prompt_eval_count/duration·eval_count/duration`을 저장,
   `SHOTS`(픽스처 경로)·`SYSTEM_MODE`(기각된 E2 재현용) env 지원.

### 6s/장(10장 1분) 도달 가능성 판정 (신선 물리 기준)

| 경로 | 추정 | 판정 |
|---|---|---|
| 32B+ollama 현 구성 | 62s/장 | 기준선 |
| 32B+vLLM(커널 2~3배 가정) | ~25~30s/장 | 미달. AWQ 재양자화로 정확도 재검증도 필요 |
| 7B/3B 계열(단독·하이브리드) | — | **정확도 불가**(파리티 실패 + 게이트 맹점) |
| OCR-primary(PP-StructureV3 등)+LLM 매퍼 | ~2~4s/장 | **유일한 후보** — 단 한글 조밀표 자릿수 정확도 미검증(스파이크 필요) |

**로컬 VLM 경로로는 6s/장은 물론 15s/장도 안전하게 도달 불가.** 남은 선택지:
(a) OCR-primary 스파이크(가장 어려운 화면 2~3장으로 자릿수 정확도부터), (b) 목표 완화(62s/장 수용,
UX를 비동기로), (c) 클라우드 API(민감정보 — 사용자가 배제).

### 운영 확인 사항 (즉시 조치 가능)

- **프로덕션 인스턴스(11434, systemd)에 드롭인이 없다** — `OLLAMA_NUM_PARALLEL=2`·`OLLAMA_KEEP_ALIVE=-1`
  미설정 상태로 서빙 중일 가능성. v2.1의 배칭 이득이 프로덕션에 실제 적용되고 있는지 불명.
  → `sudo systemctl edit ollama` → `[Service] Environment="OLLAMA_NUM_PARALLEL=2" "OLLAMA_KEEP_ALIVE=-1"`
  (실험은 올바른 env를 가진 포트 11435의 사용자 인스턴스에서 수행했다. 이중 기동 정리 필요.)

**결정: `qwen2.5-VL-32B` + `prompt4c`(압축출력) + `OLLAMA_NUM_PARALLEL=2` + 불변식 게이트**

## v2.1 — 속도 최적화 (정확도 무손실 실측: 92s/장 → 35.5s/장, 2.6배)

병목 분해가 먼저다: 화면당 92s = prefill 31~37s(이미지 4.4~5.1k tok, **compute-bound** ~145tok/s)
+ 생성 64~171s(**대역폭 포화** 6.1tok/s = 이론상한 6.5의 94% — 21GB 가중치를 토큰마다 스트리밍).
콜드로딩 89.6s는 keep_alive 만료 시마다 재발.

| 지렛대 | 실측 | 채택 |
|---|---|---|
| **출력 압축**(`prompt4c` — 지시문은 prompt4 그대로, 출력만 키 없는 positional 배열) | 생성 토큰 -43~53%(1047→603tok) | ✅ 파리티 31/31 유지 확인 |
| **디코드 배칭**(`OLLAMA_NUM_PARALLEL=2` + 화면당 요청 2개 동시 발사) | 2요청 2.37배(223→94s). 한 스텝에 가중치 1회 스트리밍으로 두 시퀀스 생성 + 한쪽 prefill이 다른쪽 디코드와 겹침 | ✅ raw 8/8 재현 동일 |
| np=4 | 2.32배 — np=2에서 이미 포화(prefill이 compute 직렬) | ❌ np=2로 충분 |
| **상시 로드**(`keep_alive:-1` + 기동 warmup) | 첫 장 -90s | ✅ |
| 2장/한 호출 | 이득 ~7s/장(프롬프트 prefill 공유)뿐 + **행→화면 귀속**이 모델 태그 의존으로 전락(finalize의 화면 경계 로직 침식) | ❌ |
| 이미지 리샘플(픽셀 축소) | **기각 실측**: ×0.66 전체 8장 → 재현율 27/31·환각 1·ISA 화면 붕괴, ×0.5 → 자릿수 10배 오류(905,940→9,059,400). 게다가 축소 화면에서 없는 열(cost)을 지어내 출력이 늘어 wall 342s로 **원본(284s)보다 느림**. 부분 사다리(2장)는 통과처럼 보였다 — 표본이 전체를 대변 못함 | ❌ |
| JPEG 품질 압축 | 무효 — 토큰 수는 파일 크기가 아니라 픽셀 치수로 결정 | ❌ |
| MoE 30b-a3b | 2.3배 빠르나 재현율 29/31(현금 2행 유실 — 게이트로 회복 불가) | ❌ |

주의(실측 함정): 두 지렛대는 **완전히 합산되지 않는다** — 압축이 디코드를 줄이면 배칭 불가능한
prefill이 새 병목이 된다(compute 직렬). 그래서 2.37×1.5가 아니라 합계 2.6배.
슬롯 수를 바꾸면 `-c = num_ctx×NP` 재적재 1회가 발생한다(run1 507s에 섞였던 그 비용) — 상시 로드로 1회뿐.

**8장 실측 284s(35.5s/장) · 재현성 raw 8/8 동일 · 파리티 PASS(기준선과 완전 동급) · 게이트 대조군 전부 통과.**
10장 ≈ 6분, 15장 ≈ 9분 (기존 24.5분 → 2.7배). 운영: `NP`(서버)=`OLLAMA_NUM_PARALLEL`(ollama) 일치 필수.
systemd 운영 시: `sudo systemctl edit ollama` → `[Service] Environment="OLLAMA_NUM_PARALLEL=2" "OLLAMA_KEEP_ALIVE=-1"`.

# (v2) 모델 선정 기록

**결정: `qwen2.5-VL-32B` + `prompt4` + 불변식 게이트** (Ollama / **Jetson AGX Orin 64GB**)
파이프라인: 비전추출 → 파서(견고화) → 게이트(합계 대조·불변식) → 증권사 정규화 → 수량 사다리(T1~T4) → 출처 표기.

목적함수도 바꿨다: 화면별 채점(`score.py`)이 아니라 **엔드투엔드 파리티**(`parity.py`) —
스크린샷 8장 → 최종 holdings를 정답표(31종목)와 대조. v1이 화면별로 100%를 받고도 파이프라인이
깨졌던 이유가 이것이다(국소 점수는 종합 결과를 보장하지 않는다).

## v1의 전제가 세 개 무너졌다

| v1의 기술 | 실측 |
|---|---|
| "Orin 32GB · 32B는 메모리 3배라 불필요" | **64GB**(가용 48GB). 32B q4 ≈ 20GB — 넉넉히 적재. 근거 소멸 |
| "32B는 평가기 OOM으로 미실행" | 그 평가는 *16GB 기계*에서 했다. Orin에서 실행하니 **32B가 유일한 통과 모델** |
| "7B가 재현율·정확도 100%" | 화면별 채점 기준이었다. 엔드투엔드로는 7B가 값 1건을 계속 틀린다 |

v1이 "남은 검증"으로 미뤄둔 세 항목을 이번에 측정했다:
①속도 33s/장(7B)·92s/장(32B) ②환각저항 → 게이트 대조군으로 검증 ③**재현성 → 3회 반복 분산 0**
(이번 회귀의 원인은 비결정성이 아니라 JSON 파싱 버그였다. 아래.)

같은 계열의 체급 상승(2.5-7B → 2.5-32B)만이 실제 이득을 냈다.

## 회귀 2건의 진짜 원인 (다음 사람이 같은 함정을 피하도록)

1. **해외주식 10종목(128M) 통째 유실** — 모델·컨텍스트 문제가 아니었다.
   모델이 화면 표기 그대로 `"cost": +263504`를 출력했고 **선행 `+`는 JSON 문법 위반**이라
   `json.loads`가 배열 전체를 거부 → 0행 → `classify([])`가 기본값 `detail` 반환 → 조용한 빈 화면.
   → 파서 견고화(부호·쉼표·트레일링콤마 교정 + **행 단위 salvage**) + 빈 화면은 `empty`로 경고.
2. **삼성 ISA → 한국투자증권 오정규화** — 오염원은 **프롬프트**였다.
   스키마의 자리표시 문자열 `"broker":"증권사명"`을 모델이 그대로 뱉자, `resolve_broker`가 그것을
   브랜드로 오인해 네이버에 "증권사명 어느 증권사"를 검색하고 **검색결과 최빈 증권사를 정답으로 채택·캐시**했다.
   → 스키마에서 자리표시 제거(값 없으면 null) + 자리표시·일반명사는 검색 금지 + **근거 없으면 답하지 않음**
   (LLM이 UNKNOWN이면 최빈 폴백도 하지 않는다) + 캐시 자가치유.

## 파이프라인이 모델 오류를 흡수한다 (모델 무관 안전망)

7B 기준 **환각 7건 → 0건, 수량 21/30 → 31/31**을 *모델을 바꾸지 않고* 회복시킨 불변식들:

- **회계 항등식** `평가금액 = 매수금액 + 평가손익` — 깨지면 지어낸 매수금액으로 보고 재계산.
  (매수금액 열이 없는 화면에서 모델은 손익을 원가 자리에 넣거나 값을 발명한다.)
- **화면 수량 검증** — 캡처일 종가로 설명되지 않는 수량은 기각하고 사다리로 재도출.
  (수량 열이 없는 화면에서 모델이 평가손익을 수량 칸에 넣는다: `qty=998400`.)
- **합계행은 이름이 아니라 산술로** 판정(어떤 행의 값 = 나머지 합 → 그 행은 총액).
  이름 목록에 기대면 '외화예수금' 같은 정당한 현금 보유행까지 지워진다.
- **한 자산 = 한 행** — 같은 금액 반복(예수금 당일/D+1/D+2)은 병합, 외화 잔액의 달러·원화 두 표기도 병합.
- **잔고는 음수일 수 없다 / 잔고 == 손익이면 손익 복제** → 오염된 총액은 대조 기준에서 제외
  (오염된 자[尺]로 대조하면 게이트가 거짓 경보를 낸다).
- **계좌 잔고 행에는 수량이 없다** → 수량 칸의 숫자는 잘못 담긴 잔고다.
- **증권사 상속은 계좌 단위**로만. '증권사가 하나뿐이면 그걸 쓴다'는 폴백은 **금지**(교차오염).
  라벨 없는 화면은 *같은 요약화면에 대조되는 화면들과 같은 앱*이라는 관계로 소속을 추론한다.
- **심볼 검증 게이트** — 이름 검색이 엉뚱한 종목을 집으면(‘메타 플랫폼스’→국내 메타랩스) 캡처일 종가로
  화면 단가가 설명되지 않는다 → 심볼 채택 안 함(조용한 오매칭 차단).
